ARM-芯片架构简介

Posted on 2022-09-05

简介

ARM(Advanced RISC Machine)架构。是一个精简指令集(RISC,Reduced Instruction Set Computer)处理器架构家族，具有低成本、高性能、低耗电的特性。
ARM架构广泛应用在嵌入式系统，现在也有应用在个人计算机（Apple M1）和超级计算机中。

  • 与ARM同属精简指令集架构的还有RISC-V，MIPS，Xtensa等。
  • 与精简指令集相对应的是复杂指令集（CISC,Complex Instruction Set Computer），代表架构为X86。

ARM架构范畴

ARM架构的主要范畴整理如下：

条目 说明
指令集 每条指令的功能；指令如何在内存中表示(它的编码)
寄存器组 有多少寄存器；寄存器的大小；寄存器的功能；寄存器的初始状态
异常模型 不同的权限级别；异常的类型；从异常中获取或返回时会发生什么
内存模型 内存访问是如何排序的；缓存是如何运作的，什么时候以及软件必须如何执行显式维护
调试、跟踪和分析 如何设置和触发断点；跟踪工具可以捕获什么信息，以什么格式捕获
安全性扩展(TrustZone) 如何设置和触发断点；跟踪工具可以捕获什么信息，以什么格式捕获

ARM-芯片启动流程

Posted on 2022-09-05

芯片启动流程

在MCU(CPU)上执行C语言编译的可执行文件时，基本可以将启动流程分为以下3个阶段：

  • 初始化阶段
  • 执行阶段
  • 退出阶段

ARM芯片也不例外，仅与其他架构芯片有一些细节上的不同。

初始化流程

硬件初始化

硬件初始化通常在启动代码cstartup.s中执行，如果需要，还可以通过用户提供的额外低级接口__low_level_init来执行用户自己的代码。它可能包括重置/启动其余硬件，设置CPU等，以准备软件c/c++系统初始化。

  • 当cpu复位的时候，，程序将从系统启动代码中的程序入口entry开始执行。
  • 堆栈指针初始化为 CSTACK 块的末尾。
  • 对于Arm7/9/11、Cortex-A和Cortex-R器件，异常堆栈指针初始化为每个相应部分的末尾。
  • 如果用户定义了函数__low_level_init，则为应用程序提供执行早期初始化的机会。

c/c++软件系统初始化

通常，此步操作确保在调用 main 函数之前，每个全局（静态链接）的 C/C++符号都会收到其正确的初始化值。

  • 静态和全局变量初始化。也就是说，清零初始化为零的变量，并将其他初始化变量的值从 ROM 复制到 RAM 存储器。 如果__low_level_init返回零，则跳过此步骤。
  • 在标准 C 中，所有静态变量（在固定内存地址分配的变量）必须由运行时系统初始化为应用程序启动时的已知值。此值是分配给变量的显式值，或者如果没有给出值，则将其清零。在编译器中，此规则有例外，例如声明为__no_init的变量，它们根本没有初始化。
    • 对于已经初始化的数据，从ROM复制到RAM
    • 对于未初始化的数据，用零填充
  • 构造静态 C++ 对象。
  • 调用主函数，启动应用程序。

应用程序初始化

就是用户自己的程序部分。这完全取决于用户的应用程序：对于带操作系统的应用程序，它可以包括设置RTOS内核并启动RTOS驱动的应用程序的初始任务；对于裸机应用程序，它可以包括设置各种中断，初始化通信，初始化设备等。
对于基于 ROM/flash 的系统，常量和函数已经放在 ROM 中。 放置在 RAM 中的所有符号必须在调用main函数之前初始化。链接器已将可用RAM划分为变量，堆栈，堆等的不同区域。

  • 当应用程序启动时，系统启动代码首先执行硬件初始化，例如初始化栈顶指针以指向预定义堆栈区域的末尾：

执行阶段

这里的执行阶段指的是应用程序的执行。
应用程序可以是无操作系统，也可以是有操作系统。

退出阶段

大多数情况，应用程序是在一个循环中执行，不涉及退出。
如果一定要中途退出main函数，有两种方法：

  • 在main函数中执行return；
  • 执行exit()；

FreeRTOS源码精读-tasks

Posted on 2022-09-05

阅读原则

tasks.c是FreeRTOS的核心代码，虽不算非常复杂，也有5000多行代码，一行行代码看的话一天也不够用。我推荐的阅读步骤是：

  • 先看API、数据结构、全局变量
  • 再看主要函数的源码
  • 最后看次要函数的源码

关键API

xTaskCreate

功能：申请动态内存，并创建任务，为任务结构体（TCB_t）赋初值。
这个函数仅在支持动态内存申请的宏使能时生效。

值得注意的是，有的CPU的栈内存是从低向高增长，有的CPU栈内存是高向低增长（如ARM架构）。这会影响任务控制块内存（pxNewTCB）和任务内存（pxStack）的申请顺序，对于一块全新的内存来说，malloc通常是优先申请低地址的内存，再申请高地址的内存。所以为了避免栈增长到溢出的时候踩踏到pxNewTCB的内存，实现了一套任务控制块内存保护机制：

  • 若CPU的内存是由高向低增长，则先申请任务内存（pxStack），再申请任务控制块内存（pxNewTCB），这样pxNewTCB就处于pxStack的更高地址，栈顶指针pxTopOfStack的初始值就放在任务内存（pxStack）的最高地址（为了向低增长），也就是内存数组的结尾处。
  • 此后，任务内存（pxStack）不断被使用，pxTopOfStack也逐渐指向更低的地址，但再也不会和任务控制块内存相遇。这就降低了自身控制块内存被自身stack内存踩踏的风险。
  • 值得注意的是，由于内存是由malloc去分配的，当内存被多次循环使用并释放后，并不能保证后申请的内存地址就比先申请的内存地址更高（也就是说，前文说提的任务控制块内存保护机制并不总能生效），也不能保证连续申请的两块内存物理地址是连续的（此时这种内存保护机制无需生效）。

xTaskCreateStatic

功能：以静态内存创建任务
这个函数与xTaskCreate的区别仅在于任务相关的内存的申请方式不通，该函数用的是预先指定的静态内存。

prvInitialiseNewTask

功能：初始化一个新的任务，被以上两个APIxTaskCreate和xTaskCreateStatic调用。

  • 重置任务内存（pxStack）为默认值，默认为0xA5
  • 根据栈增长方向设置栈顶指针pxTopOfStack的初始值
  • 初始化优先级、任务名。
  • 初始化xStateListItem，xStateListItem是一个重要的元素，其类型为一个链表的节点，挂在不同的链表（就绪，阻塞，删除，挂起）下时，意味着其处于不同的任务状态。
  • 初始化xStateListItem

vTaskDelete

功能：删除任务

prvInitialiseTaskLists

功能：初始化各个状态的任务链表全局变量，在创建第一个任务的时候会被调用。

vTaskStartScheduler

功能：开启任务调度器。

  • 创建第一个任务（或称线程），idle任务
  • （如果使能了timer宏）创建定时器任务
  • 开始调度器（xPortStartScheduler），初始化tick时钟，tick是整个操作系统的心跳。

vTaskEndScheduler

功能：结束任务调度器。（很少使用）

其他

  • 任务按相对时间延迟vTaskDelay、任务按绝对时间延迟xTaskDelayUntil；
  • 任务优先级设置vTaskPrioritySet、任务优先级获取uxTaskPriorityGet，uxTaskPriorityGetFromISR;
  • 挂起任务vTaskSuspend、恢复任务vTaskResume；
  • 挂起所有任务vTaskSuspendAll，恢复所有任务xTaskResumeAll，支持递归调用。
  • 任务上下文切换vTaskSwitchContext
  • 任务优先级继承xTaskPriorityInherit（与互斥锁有关）
  • 当前所有任务信息获取vTaskList

对任务的操作和以下几个全局变量有重要关系：

  • pxReadyTasksLists，就绪任务列表数组（数组的序号是对应的优先级）；
  • pxDelayedTaskList，延时任务列表（任务阻塞后根据阻塞时间，变成延时任务或溢出延时任务或挂起任务）；

网络协议-不同MTU的网卡如何进行TCP通信

Posted on 2022-09-04

问题

一台路由器下连接了两个子设备，一台设备的网卡MTU（最大传输单元）是1500，一台设备的MTU是1000，如何进行TCP通信，在哪一层协商？

分析

要解决这个问题，最基本的两个知识是TCP分段和IP分片，简单来说：

  • 分段特指发生在使用TCP协议的传输层中的数据切分行为
  • 分片特指发生在使用IPv4协议的网络IP层中的数据切分行为

TCP分段

TCP分段的原因是因为TCP报文段大小受MSS限制。

最大报文段长度MSS（Maximum Segment Size），为TCP传输层的最大载荷上限(即应用层数据最大长度)。
MSS是TCP的报头的可选项，MSS的协商在TCP握手阶段完成：

B将自己的MSS发送给A，建议A在发数据给B的时候，采用MSS=1420进行分段。而B在发数据给A的时候，同样会带上MSS=1372。两者在对比后，会采用小的那个值（1372）作为通信的MSS值，这个过程叫MSS协商。

通常一条TCP连接的MSS取通信双方较小的那一个MSS值，与MTU的换算关系为：
MTU = MSS + TCP首部长度 + IP首部长度
故在以太网中(网络层以IPv4为例)：
MSS = 以太网MTU - TCP首部长度 - IPv4首部长度 = 1500 - 20 - 20 = 1460字节

为什么MSS的大小需要协商呢？还得从IP层分片说起。

IP层分片

IP分片的原因是因为IP数据报大小受MTU限制。

最大传输单元MTU(Maximum Transmit Unit)，这个值是由数据链路层提供，为了告诉上层IP层，自己的传输能力是多大。
受各种因素限制，一个网卡能发出的单包数据长度是有限的，这个单包数据的最大长度就是数据链路层的传输能力。

发送端的IP分片

在网络数据发送者开始发送时，IP层就会根据MTU进行数据包切分，一般MTU=1500 Byte。

  • 假设IP层有 <= 1500 Byte 需要发送，只需要一个IP包就可以完成发送任务；
  • 假设IP层有 > 1500 Byte 数据需要发送，需要分片才能完成发送，分片后的 IP Header ID 相同，同时为了分片后能在接收端把切片组装起来，还需要在分片后的IP包里加上各种信息。比如这个分片在原来的IP包里的偏移offset。

传输路径上的IP分片

我们知道在因特网通信过程，网络数据在很多路由器节点之间跳转，而各个节点间MTU可能不同，这就导致数据是可能被多次分片的。而且每次分片都要加上各种信息便于在接收端进行分片重组。
如果有办法知道整个链路上，最小的MTU是多少，并且以最小MTU长度发送数据，那么不管数据传到哪个节点，都不会发生分片。
整个链路上，最小的MTU，就叫PMTU（path MTU）。
有一个获得这个PMTU的方法，叫 Path MTU Discovery。
Path MTU Discovery的大致原理是：

通过向对端发送申明不要分包(DF)的IP报文，然后根据是否收到传输路径上节点的“ICMP不可达报文”，来判断目前的发送数据长度是否过长，如果过长则减少长度重新试探，直至发送成功。

小结

不同MTU的两个网卡设备进行TCP通信时，两端先根据自身的MTU计算出MSS，然后在TCP握手的时候进行MSS协商。

木兰同时受到中西方思想的影响，两种文化的碰撞塑造了她。她热爱电影，钟表这些新事物，喜欢游历，却也对人情事故得心应手。她最终压制自己的爱情，接受命运。曼娘，银屏是旧社会婚姻制度的牺牲品。莫愁相对中庸，却最有福气。立夫是典型的寒门贵子，处事得体，胸有大志，未来可期。

下篇

故事结束于木兰的得道，而非主角的生死或是战争的终结，这是让我意外的。后查阅到此书封笔于三十年代末，那时抗日战争尚未结束，这便很合情合理了。
从八国联军侵华到抗日战争，本书几乎是一部中国的近代史。家庭的命运因为国家的命运而改变，每个人都不可避免地被卷入到这一段段社会巨变中。从一段段故事中我们得以从另一个视角去看待义和团运动，辛亥革命，新文化运动，五四运动，北伐战争这些历史事件，去看民众是如何看待和参与这些事件，以及事件对国家对人民产生了什么样的影响。
从看似平静的笔触中，我们可以体会到作者对于国家破败的痛心疾首。另一方面，作者又以老庄的哲学思想去看待这些历史变迁，同时对抗战胜利抱有坚定的信心。正如文中所写，不管风云怎么变幻，老北京一直在那里。当然中国大地也会一直在那里，中国的老百姓会一直在那里。所有的中国群众凝聚在一起，会成为不可战胜的强大力量。
姚老说过，他的生命会在女儿木兰和他的孙子们身上得以延续。而木兰对道的领悟则到达了另一层，她相信她的生命更会在千千万万的中国群众身上得以延续。

总结

总的来说，这部作品可以称得上是一部鸿篇巨著，其中的哲学思想也和生活紧密结合。我虽只能管中窥其一二，也觉得酣畅淋漓。

作者：保罗·柯艾略
阅读时间：2018.07.09

如果说有书能催生源源不断的精神力量，那么这本书应该是其中之一。
书中讲述了一个西班牙的牧羊少年San Diego，为了追寻梦境中宝藏前往金字塔的历程。在旅途中男孩历尽千辛万苦，通过了对勇气、智慧、执着的考验，最终收获宝藏。
少年后来意识到，真正属于自己的宝藏，其实就是这段寻找的经历。

我从这本书中学到的最深刻的事情便是，“learn from everything”，这世上的一切事物，都可以成为你的老师。
羊群可以是你的老师，风可以是你的老师，路人可以是你的老师。只要拥有足够的洞察力，便可以从宇宙万物中学习。
“鱼儿在跳出水面之前不会知道自己生活在水中”，年轻的时候去经历不同的人生，尽力打开自己的视野，是难能可贵的事情。
